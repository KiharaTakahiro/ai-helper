# コーディング規約 (スタイルガイド)

このリポジトリで遵守すべき基本的なコーディング規約です。主に Python を対象とし、英語・日本語コメントの書き方についても触れています。

## 全体方針

- [PEP 8](https://peps.python.org/pep-0008/) を基本とする。
- 40 文字超えのコメントやドキュメントは日本語を使用し、簡潔に記述する。
- 複雑な処理や設計意図は docstring に記述し、Sphinx 風のスタイルも許容。
- 英語は API や外部に公開するドキュメントのみ使い、ソース内コメントは原則日本語。

## ファイルとモジュール

- モジュール名・パッケージ名は `snake_case`。
- クラス名は `PascalCase`。
- テストファイルは `tests/test_*.py` 内に配置。

## コメントとドキュメント

- 関数/クラスの説明は docstring に書き、できるだけ詳細な情報を記述する。
  - 引数 (`Args:`)、戻り値 (`Returns:`)、例外 (`Raises:`)、処理の流れなどを含める。
  - 3 行以上にわたる場合は日本語を用い、全体的に厚めの説明を書いてください。
- 単一行コメントは `#` の後に半角スペースを入れ、日本語で説明する。
- **すべてのコメントは日本語で記述すること。テストコードも含め、プロジェクト内の
  あらゆるソースファイルが対象です。コメントが不足している箇所については
  必要に応じて追記し、他の開発者が振る舞いを理解しやすくしてください。**
- テストコードについても同様に、**各テストケース冒頭に検証する観点と狙いを明記**する。
  これによりパイプライン全体を複数の観点でカバーし、結果的に行カバレッジ100%
  を達成することを目指す。
- トップレベルの README や設計資料は自由に日本語で記載。

## 文字コード

- ソースコードはすべて UTF-8 で保存。
- README などのドキュメントも UTF-8 にする。

## 依存関係管理

- Poetry を用いて `pyproject.toml` にすべて記載。**絶対に `pip install` を直接使わないこと。**
- ライブラリ追加は `poetry add` または `poetry add --dev` を使用。
- 別のプロジェクトからパッケージをコピーした場合でも、`poetry add` で同期する。

## フォーマッタ・リンター

- 必要に応じて `black`/`ruff` などを導入して自動整形。
- CI で `ruff check` や `black --check` を実行すると効果的。

## その他

- インポートは標準ライブラリ、サードパーティ、自プロジェクトの順番で行い、空行で区切る。
- 不要な import は削除し、IDE の自動整理を活用する。
